Oop isn’t weird Grupert

2023-05-04 ::
#python  #OOP 

Yeah, it isn’t THAT weird⌗

OOP stands for “Object Oriented Programming” and it is a programming paradigm in which you write your code into encapsulated blocks of code called classes which are basically blueprints that determine how objects behave.

Objects are then bits of code that have a clear purpose and structure and they can access specific data and methods. Let’s think of an everyday object, a door. A door has many attributes, for example it’s color, it’s material, maybe it’s weight, it also has a property, is it open or closed, and it has a method, open or close. Let’s compare two paradigms, procedural vs OOP with this example:

procedural⌗ 

def make_door(name, color, material):
    door = {
        'name':name, 'color': color, 'material': material, 'status': 'closed'
    }

def open_door(door):
    door['status'] = 'opened'

def close_door(door):
    door['status'] = 'closed'

def is_door_open(door):
    return door['status'] = 'opened'

my_door = make_door('front door', 'red', 'wood')

OOP⌗ 

class Door:
    def __init__(name, color, material):
        self.name = name
        self.color = color
        self.material = material
        self.status = 'closed'
    
    @property
    def is_opened(self):
        return self.status == 'opened'
    
    def open(self):
        self.status = 'opened'
    
    def close(self):
        self.status = 'closed'

my_door = Door('front door', 'red', 'wood')

As you can see both codes are easily legible and maintainable. OOP encapsulates all the logic pertaining to an object within it’s class. Procedural defines blocks of selfcontained logic that are reusable and minimalistic. And both are adecuate when programming. But then, what’s the point of this overly complicated OOP stuff? Well, let’s say you want to expand the code, so that this is not just some abstract weird door, but a part of a car. In OOP it might be as simple as extending the class thorough inheritance. Inheritance means that a class gets some base parameters from another class, and then you can expand on it, say we just want to add an electrical window to our door.

class CarWindow:
    ... # we define the logic for the window here


class CarDoor(Door):
    def __init__(self, *args, **kws):
        # this just means "do whatever you were doing within the Door class"
        super().__init__(*args, **kwargs)
        # and then add the new property
        self.drivers_window = CarWindow(...)

Above we see something new, super what it does in simple terms is executing the method of the base class, in this case Door. So what we’re doing is simply doing the same thing Door is doing but also adding a driver’s window, which is a class istance itself. A subclass can override methods or attributes of the base class if needed to change how they work or extend their functionality.

Let’s see another example of this simple inheritance.

class Animal:
    def __init__(self, name):
        self.name = name
    
    def talk(self):
        raise Exception('talk methos has not been implemented')

class Dog(Animal):
    def talk(self):
        print('woof')

class Cat(Animal):
    def talk(self):
        print('miau')

In here we’re not doing any weird things, we define a base class, with a method talk which raises an exception because it’s not implemented yet. This kinds of classes are called “base abstract classes” or “ABC” for short, and they define how a class should look like, but are not intented to make objects by themselves but instead they’re meant to be inherited from. So now, we have a Dog and a Cat that are both animals and both can “talk”, this is called polymorphism and is a key concept in OOP, it simply means that we can use the same method on multiple classes when they inherit from a base class, so we can have any animal “talk” it doesn’t matter if its a dog, a cat or mouse, as long as they’re animals, they’ll have a talk method.

And now it’s a very good time to review one of the best advantages of OOP, which is types. Imagine you’re an inventor (which if you code you kind of already are) and you invented a washing machine which can wash cats, dogs, and yes cars. Since you don’t want to kill your pets, you have to identify the method to use for washing them vs washing the cars. How you’d do it with OOP?

if isinstance(item, Animal):
    machine.wash_animal(item)
elif isinstance(item, Car):
    machine.wash_car(item)

Now think about how you’d implement this in another programming paradimg. You would first need a way to determine if the item is an animal or a car and this will implicitly add more complexity to your code, becase for every new item your machine can process, you’ll need a function to check if it’s the correct kind. This doesn’t mean it can’t be done or that it’s overly complex, but OOP excels at this kind of situations.

Any programming project must be well organized, easy to read and maintain. This is true for any paradigm and it will be different for every one, but OOP is awesome to simplify complex checks, encapsulate and manipulating data and modeling data structures.

So don’t be afraid of OOP, it’s awesome!
